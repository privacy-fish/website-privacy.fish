privacy.fish exists because private and secure access to the internet is a human right.

An email address is now basic internet infrastructure. It is required for banking, work, travel, software accounts, public services, and almost every modern online identity. That makes email too important to treat as another advertising surface or behavioral data source.

## Privacy before convenience

Most email services optimize for convenience first. They keep mail online indefinitely, provide browser access, make recovery easy through identity checks, and add bundled services that increase the amount of data and code involved.

privacy.fish makes the opposite tradeoff. We would rather ask users to adapt their workflow than ask them to trust us with more information than we technically need.

That means:

- no webmail,
- no password login,
- no analytics or tracking,
- no unnecessary retained account data,
- no provider-held mailbox access,
- no recovery path based on personal identity.

These choices are less convenient, but they reduce retained data, attack surface, and trust.

## Security before features

Every feature is also a liability. A larger system has more code paths, more dependencies, more logs, more permissions, and more ways to fail.

privacy.fish is intentionally narrow. It focuses on email access through SSH / SFTP, local device keys, encrypted server-side storage, and short retention windows. We avoid convenience features when they require storing more data, exposing more interfaces, or increasing the amount of provider-side trust.

## Data minimization by default

The safest personal data is data we never collect.

When data is technically required, it should be retained for as little time as possible. When it is no longer required, it should be deleted rather than endlessly protected. This is why the service is designed around short mail retention, temporary payment references, manual provisioning, and minimal operational records.

## Open infrastructure

Security claims should be inspectable.

privacy.fish publishes the software and infrastructure code used to build and administer the service. The goal is not to ask users to blindly trust private claims, but to make the design available for review, criticism, and improvement.

## The principle

privacy.fish is not trying to be the easiest email service.

It is trying to be an email service for people who value privacy and security enough to accept a different workflow.
